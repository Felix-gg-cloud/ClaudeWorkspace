# 风险与缓解措施（Risk & Mitigation）

本文档针对 **AI-Only 校验模式**下的主要风险进行分析，并给出对应的缓解策略。

---

## 一、风险总览

| # | 风险                     | 影响                                           | 可能性 | 严重程度 |
|---|--------------------------|------------------------------------------------|--------|---------|
| 1 | 输出不一致（非确定性）      | 同一数据在不同时间校验结果不同，难以审计            | 中     | 高      |
| 2 | 规则漏判或误判             | AI 遗漏规则或发明规则目录外的校验，导致报告不可信    | 中     | 高      |
| 3 | Token 超限                | 大文件单次调用超过模型上下文窗口，调用失败           | 高     | 中      |
| 4 | 响应延迟 / 超时            | LLM 推理耗时，Flow 超时失败                       | 中     | 中      |
| 5 | 输出格式不稳定             | AI 返回非合法 JSON，Power Automate 解析失败        | 低     | 中      |
| 6 | 成本控制                  | 大量 token 调用产生较高 API 费用                   | 中     | 低      |

---

## 二、逐项风险与缓解措施

### 风险 1：输出不一致（非确定性）

**描述**：默认配置下，LLM 输出存在随机性，同一行数据可能在不同运行中得到不同的校验结论。

**缓解措施**：

| 措施                       | 实施方式                                                                                   |
|--------------------------|------------------------------------------------------------------------------------------|
| 设置 `temperature = 0`   | 在 Copilot Studio 或 Azure OpenAI 调用参数中将 `temperature` 设为 0，使输出接近确定性         |
| 输入哈希校验               | Power Automate 对每批次行数据计算 MD5/SHA-256 哈希，同批次哈希相同时复用上次结果，跳过重复调用 |
| 抽样回归测试               | 维护一组标准测试数据（golden dataset），每次主版本更新后运行抽样测试，对比输出是否一致          |
| 明确提示词约束              | 在 System Prompt 中明确写明"对同一输入你的输出必须一致"，并要求严格执行规则目录               |

---

### 风险 2：规则漏判或误判

**描述**：AI 可能遗漏规则目录中的某条规则，或基于"语义推断"发明规则目录之外的校验，导致误报或漏报。

**缓解措施**：

| 措施                          | 实施方式                                                                                       |
|-----------------------------|----------------------------------------------------------------------------------------------|
| 规则目录内嵌 System Prompt    | 将完整规则目录硬编码在 System Prompt 中，而非依赖模型记忆                                         |
| 明确禁止发明规则               | System Prompt 中加入约束："不得自行发明或添加规则目录之外的校验规则"                               |
| 抽样人工审核                   | 每月随机抽取 5-10 行已知正确/已知错误的样本，与 AI 输出对比，验证规则执行准确性                    |
| POC 阶段并行运行              | POC 期间同时保留一份确定性 Power Automate 规则逻辑，与 AI 输出进行对比，发现偏差后及时修正 Prompt  |

---

### 风险 3：Token 超限

**描述**：`tblOffshoring` 数据量增长时，单次请求超过模型上下文窗口（如 16k / 128k tokens）导致调用失败。

**缓解措施**：

| 措施                          | 实施方式                                                                                       |
|-----------------------------|----------------------------------------------------------------------------------------------|
| 分页读取 + 批量调用             | Power Automate 使用 `Top Count` + `Skip Token` 分页，每批限制行数（建议 ≤ 50 行/批）            |
| 按 YearMonth 分批             | 预先按月份分组，每个月份数据作为独立批次调用，天然控制批次大小，且便于按月存储报告               |
| 动态批次大小                  | 根据每行平均字符数估算 token 数，动态调整 `varPageSize`（参考：1 token ≈ 4 字符）                |
| 精简 payload 字段             | 仅传送 rules engine 需要的列，去掉无需校验的附加元数据列，降低每行 token 消耗                    |

**Token 估算参考**：

| 模型           | 上下文窗口   | System Prompt 占用（约）| 可用于数据行（约） | 建议每批行数 |
|--------------|------------|----------------------|-----------------|------------|
| GPT-4o       | 128k token | 2k token             | 126k token      | 200-300 行  |
| GPT-4-turbo  | 128k token | 2k token             | 126k token      | 200-300 行  |
| GPT-3.5-turbo| 16k token  | 2k token             | 14k token       | 30-50 行    |

---

### 风险 4：响应延迟 / 超时

**描述**：LLM 推理耗时可能超过 Power Automate HTTP Action 的默认超时（120 秒）。

**缓解措施**：

| 措施                          | 实施方式                                                                                     |
|-----------------------------|--------------------------------------------------------------------------------------------|
| 控制批次大小                  | 每批 ≤ 50 行，减少单次推理 token 数量，缩短响应时间                                           |
| 异步调用模式                  | 使用 Azure OpenAI 异步 API（`202 Accepted` + polling），避免同步等待超时                       |
| 设置合理超时                  | Power Automate HTTP Action `Timeout` 设为 120 秒，并配合 Retry Policy（最多 2 次）             |
| 批次并行执行（高级）           | 使用 Power Automate Premium 的 `Apply to each` 并发度设置，多个 YearMonth 批次并行调用         |

---

### 风险 5：输出格式不稳定

**描述**：AI 返回非合法 JSON，或在 JSON 外包裹额外说明文字，导致 Power Automate `Parse JSON` Action 失败。

**缓解措施**：

| 措施                          | 实施方式                                                                                     |
|-----------------------------|--------------------------------------------------------------------------------------------|
| 明确输出约束                  | System Prompt 中写明："返回内容必须是合法 JSON，外层为 `{ issues, report }`，不得在 JSON 外添加任何文字" |
| 使用 JSON mode               | Azure OpenAI API 支持 `response_format: { type: "json_object" }`，强制返回合法 JSON           |
| Power Automate 容错处理       | 使用 `Configure run after` + Scope 捕获解析异常，记录原始响应到 SharePoint 审计日志，不中断整体 Flow |
| 输出 Schema 验证              | 使用 `Parse JSON` Action 配合 JSON Schema，解析后验证 `issues` 为数组、`report` 为字符串         |

---

### 风险 6：成本控制

**描述**：大量行数据通过 LLM 处理，产生较高 API token 费用。

**缓解措施**：

| 措施                          | 实施方式                                                                                     |
|-----------------------------|--------------------------------------------------------------------------------------------|
| 增量校验                      | 仅对上次校验后**新增或修改**的行调用 AI 校验，已通过的行复用上次结果（结合行哈希实现）            |
| 选用合适模型                  | POC 阶段优先使用 `GPT-3.5-turbo-16k`（成本约为 GPT-4o 的 1/10），规则执行类任务效果基本满足需求 |
| 监控 token 消耗               | 在 Power Automate 中记录每次调用的 `usage.total_tokens`，设置月度预算告警                      |

---

## 三、总体建议

### POC 阶段（当前）

1. **并行运行**：保留现有确定性 Power Automate 规则作为参照基准，与 AI 输出进行每月对比。
2. **小规模验证**：先用 1-2 个月的历史数据（已知正确结果）验证 AI 输出准确性，再扩展到全量数据。
3. **人工抽样**：每次运行后，人工检查 10% 的 issues，确认 AI 输出符合规则目录预期。

### 生产化阶段（POC 通过后）

1. 去掉确定性并行逻辑，全量切换 AI-Only 模式。
2. 建立自动化回归测试套件（golden dataset）。
3. 将规则目录版本化管理（Prompt 版本 + Git tag），确保规则变更可追溯。
4. 配置 Azure OpenAI 的内容过滤和 API key 轮换策略。
